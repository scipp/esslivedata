1What this is for

The broad picture: more and more streams need to feed values into Sciline workflow graphs as parameters (not as accumulating data). Each of the following cases has the same shape — a stream's latest value parametrises a workflow graph for one or more sources — but historically each used different plumbing.

CaseStream value drivesStatus
f144 / NXlog integration Generic F144Stream + device synthesis Recently landed on main
Bifrost rotation InstrumentAngle[SampleRun], SampleAngle[SampleRun] on cut workflows In this branch (instrument-scope, skip_motion on non-consumers)
LOKI rear-bank carriage NeXus depends_on transformation patched via add_dynamic_transform In this branch (instrument-scope with transform_path, skip_motion on non-consumers)
ROI (rectangle / polygon) Per-job context binding from the dashboard In this branch (spec-scoped with resolver + seed)
Chopper setpoint / phase Wavelength frame, TOA → λ lookup Coming next — this infrastructure is its prerequisite

The branch lands one mechanism that covers the first four uniformly and unblocks the fifth. The runtime-behaviour cleanup (ADR 0002) is a side effect of taking these wirings seriously: once "context bindings" are first-class, gating on their availability is a small predicate.

2The mess being cleaned up

Three structural problems on main, each from a different attempt to bolt a new wiring onto existing carriers:

1. AuxSources carrying two unrelated concerns

AuxSources was built for user-facing dynamic-aux selection (LOKI's incident / transmission monitor choices). Context streams started piggy-backing on it for routing — declared as plain-string aux entries that the UI happens to ignore. ROI did this via DetectorROIAuxSources; bifrost rotation did it via bifrost_aux_sources. A routing primitive ended up doubling as a workflow-execution primitive.

2. The same stream declared in multiple places

For LOKI's detector_carriage: one dict (LOKI_DYNAMIC_TRANSFORMS in specs.py) passed to two consumers — DetectorROIAuxSources(dynamic_transforms=…) for routing and DetectorViewFactory(dynamic_transforms=…) for graph wiring. Names must agree across the call sites; nothing enforces it.

For ROI: 'roi_rectangle' and 'roi_polygon' appear independently in DetectorROIAuxSources.inputs (routing) and detector_view/factory.py's context_keys (graph wiring). Two places, manual sync.

3. No spec-scope declaration site for context bindings

LogContextBinding on Instrument was the only first-class context-binding type, source-scoped and f144-specific in name. Bifrost rotation used it. Anything spec-scoped — ROI, LOKI carriage — had no comparable home and reached for AuxSources subclasses, propagating the structural problems in #1 and #2.

Adding chopper setpoint / phase streams next would have meant either another AuxSources subclass or inventing a third carrier. ADR 0003 makes that unnecessary.

3The declaration model (ADR 0003)

One record. Two declaration sites. One JobFactory merge.

The record

@dataclass(frozen=True, slots=True, kw_only=True)
class ContextBinding:
    stream_name:        str                                       # workflow-facing
    workflow_key:       Any = None                                # Sciline key; inferred for chain-patch
    dependent_sources:  frozenset[str]
    transform_path:     str | None = None                       # NeXus path for chain-patch bindings
    stream_resolver:    Callable[[JobId, str], str] | None    # default: identity
    seed_factory:       Callable[[JobId], Message]   | None    # default: no seed

Renamed from LogContextBinding (the f144-specific predecessor). transform_path carries the NeXus chain-patch target so it no longer lives in a separate per-instrument dict; when set, workflow_key defaults to TransformValueLog (resolved lazily by JobFactory.create). Construction raises ValueError if neither field is set. src/ess/livedata/config/stream.py:92

Two scopes

Instrument scope

Instrument.add_context_binding(...) in factories.py.

Activates for every spec whose source set overlaps dependent_sources. No resolver, no seed. Specs that consume the source but not the value opt out via SpecHandle.skip_motion() — a single boolean, coarse but simple. The instrument-scope default makes silent omission impossible: a new consuming spec picks up the binding by default, and forgetting to skip_motion on a non-consumer produces a noisy gate-on-unused-stream rather than silently-wrong output.

Today: LOKI carriage (chain-patch via transform_path) and bifrost's two rotation streams (direct-bind).

Spec scope

SpecHandle.add_context_binding(...) in factories.py.

Defaults dependent_sources to frozenset(spec.source_names). Supports stream_resolver (for per-job wire names) and seed_factory (for cold-start defaults). Not subject to skip_motion — if the spec declared it, the spec wants it.

Today: ROI (resolver + seed).

Late-binding from factories.py keeps workflow-key imports out of specs.py. That invariant exists because the dashboard imports specs.py to render the UI and must not pull in instrument-specific Sciline pipelines.

How JobFactory merges them

def create(self, *, job_id, config) -> tuple[Job, list[Message]]:
    instrument_inputs = [] if spec.skip_motion else instrument.context_bindings
    matching = [ci for ci in (*instrument_inputs, *spec.context_bindings)
                if job_id.source_name in ci.dependent_sources]

    context_keys         = {ci.stream_name: _resolve_workflow_key(ci) for ci in matching}
    transform_paths      = {ci.stream_name: ci.transform_path for ci in matching
                            if ci.transform_path is not None}
    wire_for             = {ci.stream_name: (ci.stream_resolver(job_id, ci.stream_name)
                                             if ci.stream_resolver else ci.stream_name)
                            for ci in matching}
    rendered_aux_names   = {**rendered_aux_names, **wire_for}      # splice into aux
    context_stream_names = set(wire_for.values())                # strict subset of aux
    seed_messages        = [ci.seed_factory(job_id) for ci in matching
                            if ci.seed_factory is not None]

    # Seed messages are forwarded by JobManager.schedule_job to the preprocessor.

Context wire names splice into aux_source_names, so existing Job._filter_data_for_job and _stream_to_fields routing handles both dynamic and context bindings uniformly. The gate set lives in its own field. _resolve_workflow_key defaults chain-patch entries (those with transform_path set) to TransformValueLog. src/ess/livedata/core/job_manager.py:138

Routing pickup (a hard co-requirement)

gather_source_names learns to include context-binding streams in the per-namespace Kafka subscription set:

  • Instrument-level entries — picked up when any spec in the namespace shares a source with the binding's dependent_sources.
  • Spec-level entries — picked up only when stream_resolver is None. Job-scoped entries (ROI) materialise per-job wire names that can't be resolved at namespace-startup time; they route via a dedicated topic instead (today StreamKind.LIVEDATA_ROI).

Without this extension the bifrost rotation and LOKI carriage f144 streams would never be subscribed and the gate would stay closed indefinitely. src/ess/livedata/config/route_derivation.py:29

4The runtime invariant (ADR 0002)

A small, orthogonal piece. Once context bindings are first-class declarations, gating their availability at the JobManager becomes a single predicate.

push_data(WorkflowData) ┐ │ ▼ ┌─────────────────────────────────┐ │ _gate_pending_context(available)│ │ │ │ for each active job: │ │ seen ∪= available ∩ aux │ │ missing = job.missing_context(seen) │ if missing: │ │ state = pending_context │ │ gated.add(job_id) │ └─────────────────────────────────┘ │ ▼ for job in active_jobs: if job in gated: skip # primary dropped, not buffered else: process

Key pieces

  • Job.context_stream_names: set[str] — strict subset of aux_source_names.values(). Populated by JobFactory.create from the matched ContextBinding records.
  • Job.missing_context(available) → set[str] — the predicate.
  • JobManager._seen_context_streams: dict[JobId, set[str]] — sticky per-job seen-set. Once a context value arrives it stays seen; the gate doesn't need cached context refilled every tick.
  • JobState.pending_context — new state. Distinct from warning (which means "last run produced an error") so a healthy-but-waiting job isn't confused with one that failed.
  • Seed-at-schedule: JobFactory.create returns (Job, list[Message]); JobManager.schedule_job forwards seeds to MessagePreprocessor.seed_messages and marks their wire names as already-seen.

Prior behaviour

Without a gate, the bifrost cut workflow's process_chunk_workflow reads rotation inside accumulate and crashes with 'NoneType' object has no attribute 'ndim'. Other workflows silently attribute detector data to a default geometry. The gate replaces these with JobState.pending_context plus a warning message — the workflow is not invoked until every context binding has a value.

5For authors of new specs / factories

Decision tree: which scope?

Is this a motion / geometry stream tied to a specific source?
→ Yes Instrument scope. Use transform_path for chain-patch (NeXus depends_on driven); use workflow_key for direct parameter binding. Non-consuming specs on the same source call handle.skip_motion() to opt out of the gate.
→ No, it's a property of the workflow (per-job, dashboard-driven, etc.) Spec scope. e.g. ROI (only detector views have it).

Anti-pattern: declaring motion at spec scope when more than one spec on the source consumes it — leads to forgetting one and getting silently-wrong output. Instrument scope plus skip_motion on non-consumers is the safer default.

The factory opt-in

Factories that need context_keys or transform_paths declare the kwarg; WorkflowFactory.create introspects and passes it only when present.

@handle.attach_factory()
def _factory(params: MyParams,
             context_keys: dict[str, type],
             transform_paths: dict[str, str]) -> Workflow:
    ...
    return StreamProcessorWorkflow(wf, ..., context_keys=context_keys)

Fifth opt-in kwarg now (source_name, params, aux_source_names, context_keys, transform_paths). A sixth would warrant rethinking the dispatch — possibly a single context-aware factory hook bundling all the per-job state.

6Edge cases & workarounds

Collapsed by default. Each is a small thing worth knowing but not central to the design.

designChain-patch wiring: transform_path on ContextBinding

For chain-patch motion bindings, the NeXus path of the dynamic transformation lives on ContextBinding.transform_path. JobFactory.create builds a transform_paths dict from matching records and dispatches it to factories that declare the kwarg. DetectorViewFactory iterates it to call add_dynamic_transform. workflow_key is inferred as TransformValueLog when transform_path is set.

The path is hand-typed against the NeXus file. Validating it against the actual NeXus geometry at Instrument.__post_init__ would catch typos at registration — open follow-up.

routingJob-scoped streams route via a dedicated topic, with no central registry

gather_source_names can't resolve f"{job_id}/{name}" at startup, so spec-level entries with a resolver are skipped by the route-pickup. They route via a dedicated topic — today only StreamKind.LIVEDATA_ROI.

The plumbing is hardcoded in three places per kind:

  1. StreamKind.LIVEDATA_ROI in core/message.py
  2. .with_livedata_roi_route() on RoutingAdapterBuilder, called unconditionally by services/detector_data.py:36
  3. case StreamKind.LIVEDATA_ROI: return LatestValueHandler() in handlers/detector_data_handler.py:53

A second job-scoped context kind would need all three additions. No central registry of context-stream kinds — contrast with AuxSources.inputs for dynamic aux.

validationRegistration-time collision check assumes resolver purity

Instrument._validate_context_binding_wire_name_collisions guards against the case where an instrument-level and a spec-level ContextBinding resolve to the same wire name for the same (spec, source) pair. Raises ValueError at registration.

Implementation uses the unresolved stream_name as the collision key — sound only because resolvers are assumed to be pure name-suffixing operations. Documented in ContextBinding's docstring; not enforced. A resolver that hashes or pulls from JobId metadata would make the check unsound. ROI's f"{job_id}/{name}" is the only resolver today.

optimisationSticky _seen_context_streams instead of refilling each tick

peek_pending_streams deliberately excludes active jobs' aux from the "what do I need next" set — the orchestrator does not refill cached context into WorkflowData.data every tick. Doing so would re-fire set_context on unchanged values and trigger eager downstream recompute.

Instead, _seen_context_streams[job_id] is sticky: once a context value arrives it stays seen. The seen-set widens on any aux for the job (not just context aux); that's harmless because missing_context only checks the strict subset, but it's slightly more than strictly needed.

stateJobState.pending_context chosen over overloading warning

A time-active but context-gated job is conceptually different from one whose last run errored. The new state preserves that distinction. The dashboard widget hasn't been updated to handle the new state — see Discussion item 1.

patternWorkflowFactory.create opt-in kwarg drift

Five opt-in kwargs now (source_name, params, aux_source_names, context_keys, transform_paths). The pattern still reads but is at the limit. A sixth would warrant rethinking — possibly a single context-aware factory hook with everything bundled.

dataROI seed message details

_roi_rectangle_seed and _roi_polygon_seed build messages byte-identical to what the dashboard publishes when the user deletes the last ROI: 

Message(
    timestamp=Timestamp.from_ns(0),
    stream=StreamId(kind=StreamKind.LIVEDATA_ROI,
                    name=f"{job_id}/roi_rectangle"),
    value=models.RectangleROI.to_concatenated_data_array({}),
)

Reset does not re-seed: context accumulators live in MessagePreprocessor above the job and are not cleared by Job.reset. The previously-seen value persists across reset.

testBifrost slow test asserts the drop path only

test_bifrost_qmap_drops_events_until_rotation_arrives covers gate-closes-and-drops. It does not cover gate-opens-and-emits (the recovery path) because the data-reduction test harness doesn't subscribe to motion topics for log-context bindings yet. Follow-up after the routing pickup is exercised end-to-end.

storagePydantic model_rebuild at the bottom of workflow_spec.py

WorkflowSpec.context_bindings: list[ContextBinding] uses a forward-ref; ContextBinding lives in stream.py and references JobId + Message from workflow_spec.py / message.py. To break the cycle, workflow_spec.py imports ContextBinding at the bottom and calls model_rebuild(_types_namespace={…}). Slightly awkward but localised.

compositionAux merge in JobFactory.create

Context wire names splice into rendered_aux_names ({**aux, **wire_for}) so existing Job routing handles both kinds uniformly. The gate discriminates via context_stream_names.

Implication: Job.aux_source_names now mixes routing semantics for dynamic aux (user-selectable) and context aux (declared). Worth a comment in the Job docstring for cold readers — currently absent.

scopeParam-dependent gating — explicit non-goal

Today's detector-view factory wires position-context unconditionally if the source has a motion binding, regardless of coordinate_mode (TOA vs wavelength). Over-gating accepted: TOA mode doesn't consume position but still gates. YAGNI today. If "raw detector with broken motion controller" becomes operational, the answer is to split the spec (TOA-only + wavelength variant), not to add param-dependent gating logic.

scopeAuxSources slimmed but still alive

Back to its original "user-facing dynamic-aux selection" role. initial_context_messages and DetectorROIAuxSources are gone. Bifrost qmap, LOKI xy_projection, DREAM projection no longer set aux_sources. Live usage remains for LOKI incident/transmission monitor selection.

styleNo backwards-compat shims left behind

LogContextBinding, add_log_context_binding, get_context_keys, DetectorROIAuxSources, bifrost_aux_sources, LOKI_DYNAMIC_TRANSFORMS, Workflow.context_keys, AuxSources.initial_context_messages, dynamic_transforms kwarg, SpecRequirements.requires_aux_sources — all deleted. External code on these will hard-break.

7Discussion

Dashboard pending_context signal is wrong, not just missing

The only thing in this branch that regresses user-visible behaviour vs main. Before: workflows with missing context would crash visibly (red status, error message). After:

  • STATUS_COLORS in workflow_status_widget.py:45-52 doesn't include pending_context. .get(state, default) falls back to active green. Blocked jobs render as healthy.
  • worst_state reduction (lines 784-792) walks error → warning → stopped → active. pending_context matches none of the arms; aggregation never escalates past active.
  • NICOS status export (kafka/x5f2_compat.py:103-116) falls through to NicosStatus.UNKNOWN (999) via case _. Safe but uninformative.

The warning message is set and visible in per-job detail — but the at-a-glance signal lies. Should land alongside the ADR PR or as immediate follow-up.

Resolver purity assumption in the collision check

Documented in ContextBinding's docstring; not enforced. Acceptable today (only ROI has a resolver, and it's pure name-suffixing). Consider an explicit assertion or a typed resolver protocol if more resolvers appear.

Sticky seen-set widens on all aux, not just context

Functionally fine — missing_context queries the strict subset. Slightly confusing to a cold reader of _gate_pending_context. Could narrow with available & job.context_stream_names, or just add a comment.

Opt-in kwarg drift in WorkflowFactory.create

Five kwargs today, all opt-in. The pattern still reads but is at the limit. A sixth (e.g. chopper context) would warrant rethinking — possibly a single context-aware factory hook bundling all the per-job state.

Recovery-path coverage for the bifrost slow test

Currently only asserts the drop path. Adding the rotation-arrives-and-flows assertion needs the test harness to publish motion topics; the routing-pickup extension is in place so the blocker is the harness, not the production code.

NeXus path validation

ContextBinding.transform_path is hand-typed. Validating it against the actual NeXus geometry at Instrument.__post_init__ would catch typos at registration instead of runtime. Cheap to add — open follow-up.

Job-scoped routing extensibility

A second job-scoped context kind requires: a StreamKind enum value, a RoutingAdapterBuilder.with_*_route() method called from services/detector_data.py, and a case branch in detector_data_handler.py:53. ContextBinding and the gate are generic; the per-kind plumbing is not. Decide now whether to factor a registry, or grow per-kind if a second appears.

pending_context after reset_job

Reset clears the warning and sets state to active; if context is still missing, the next tick re-enters pending_context. Correct but the brief blip is observable. Not a bug; cosmetic.

Job.aux_source_names is now overloaded

Carries both dynamic-aux and context wire names. Gate discriminates via context_stream_names. Slightly more mental-model overhead for someone reading Job cold; a comment in the Job docstring would help.
