Workflow spec consolidation

Unified plan consolidating the workflow-spec surface of ess.livedata. Covers long-term direction only insofar as it makes near-term decisions legible; near-term work is the concrete deliverable.

TL;DR

  • Consolidate in-repo — ess.livedata.schemas subpackage, pydantic+stdlib only, CI-enforced.
  • Drop scipp from the schema layer — ArraySpec replaces sc.DataArray templates; parameter models lose their sc.Variable-returning methods.
  • Reduce WorkflowId identity to (instrument, name, version); namespace leaves identity.
  • Runtime JSON-Schema announcement is the cross-service contract. No importable universal catalog.
  • Extraction to ess.reduce.schemas is deferred — gated on a stable shape and a concrete trigger before the second instrument rolls out.
  • Per-technique cross-compute unification stays opt-in, per-technique, when maintainers engage. Consolidation-in-repo is itself a complete outcome.

1 — The problem today

Today's spec surface is scattered across config/workflow_spec.py, config/instrument.py, parameter_models.py, handlers/*_specs.py, and per-instrument config/instruments/<inst>/. The concrete consequences:

Parallel interface declarations

ess.livedata maintains pydantic declarations of every workflow in parallel with what technique packages declare via sciline providers and scipp newtypes. Keeping them in step is an ongoing manual burden — drift happens without anyone acting.

Default drift

A livedata parameter default is not guaranteed to match the technique package's default for the same quantity. Scientists configure the same workflow differently in different tools.

Scipp coupling in the schema layer

Output templates use sc.DataArray default factories; parameter models carry get_edges()-style methods returning sc.Variable. Anything introspecting specs transitively imports scipp.

No UI story beyond the dashboard

Any out-of-process consumer (NICOS, future web dashboard, technique-package ipywidget UI) has nothing to import and no wire protocol to read.

Contribution overhead

Adding a parameter touches livedata, its factory, and the technique package separately. Three edits for one concept.

Principles

1. Spec vs. WorkflowImpl split

Committed

The spec (identity, param model, outputs model, display metadata) is pure pydantic and carries no sciline/scipp. The implementation (sciline pipeline, accumulators, target keys, translation layer) lives where the reduction code lives — today in ess.livedata, eventually in technique packages. Dependency arrow goes impl → spec, never the reverse.

Why this matters

Anything that lets sciline or scipp leak into the spec layer forecloses the whole point: out-of-process consumers that don't install scipp, and technique-package reuse. The reverse direction (spec importing impl) would couple the UI-facing contract to compute, which defeats the separation.

2. Translation layer next to the pipeline, never on the param model

Committed

Sciline binding is a free-function layer — apply_params(pipeline, params) and dynamic_keys(params) per workflow — co-located with pipeline construction. No Annotated[T, SclineBind(...)] on fields.

Why not field annotations?

Annotations on param fields that reference sciline types pull sciline into the schema layer. Any non-Python consumer of the schema either tolerates sciline-shaped metadata it cannot interpret, or has to filter it out. The free-function layer keeps the schema layer importable without sciline.

3. Multi-level-defaults rule

Committed

If a field exists in the param model, the param model is authoritative. The translation layer always applies it. Sciline-level pipeline[X] = value "defaults" are permitted only for structural constants — anything else is a param masquerading as a default and a silent source of streaming/batch divergence.

Why this rule now?

Without this rule, batch and streaming can silently diverge for the same param value. That is fatal for cross-compute unification. Cheap to adopt now, painful to retrofit. Project-wide discipline, not a local design choice.

4. No graph reshapes in livedata

In principle

Pins on inputs feeding declared outputs are fine (operational facts of the Kafka topic set). wf.insert(...) of replacement providers, scalar substitutions like CorrectedMonitor[EmptyBeamRun, ...] = 1.0, rebranding runs — not fine. Each is a shadow algorithm that will silently drift from the authoritative batch version.

Why "in principle", not "committed"?

The discipline kills a class of bug by construction. But it only holds if upstream (technique-package) maintainers absorb the category-C PRs that the discipline produces. If they resist, there will be temptation to reshape locally "just this once" — and that's exactly how shadow-algorithm rot wins. The rule is committed as a rule; its success is gated on external responsiveness. Listed as a falsification condition below.

5. Runtime JSON-Schema announcement is the cross-service contract

Committed

Each running service publishes the JSON Schema of its registered (reduced) workflow specs on the existing status topic. Authority lives at runtime, not in a compile-time import. Pydantic validators run backend-side; the UI gets JSON Schema for form rendering and optimistic validation.

No importable universal catalog?

Correct. There is no importable "all workflows for all instruments" package, and there is no planned ancestor for one. That assumption was the largest single source of scope blow-up in the earlier draft (fix latency, version mismatch, rollback asymmetry, cross-team on-call, dev-loop friction, release-cadence coupling — all derived from it). Dropping it removes ~70% of the plan's operational anxiety.

6. Consolidation-in-repo is a complete outcome

Committed

The schema subpackage is in-repo, framework types only, with an enforced pydantic+stdlib dependency boundary so later extraction to ess.reduce is mechanical rather than a redesign. Extraction happens when the shape has stabilised, not speculatively.

What if no second consumer ever appears?

Consolidation-in-repo is the endpoint. Not a failure. The benefits — testability, clean separation, forced design decisions, JSON-Schema emission for out-of-process consumers — all hold without extraction.

7. Per-technique ess.<tech>.schemas submodules

In principle

Cross-compute sharing is incremental. If and when a technique-package maintainer engages concretely, that package grows an ess.<tech>.schemas submodule (pydantic-only) for its technique-specific param/output shapes, importing framework types from ess.reduce.schemas.

Why "in principle"?

This is opt-in per technique, not a precondition for anything. Technique-package maintainers are stakeholders we don't control. If none engage, the plan still stands.

3 — What's in, what's out

The strictest constraint on the plan. Anything on the right is excluded by design, for reasons detailed in the principles.

In This branch's deliverables

  • Core spec types: WorkflowSpec, WorkflowId, WorkflowOutputsBase, AuxInput, AuxSources.
  • Output templates: ArraySpec / CoordSpec replace sc.DataArray.
  • Instrument identity: InstrumentSpec, ComponentSpec.
  • Common parameter building blocks: wavelength / TOF / two-theta / energy ranges, EdgesModel variants, unit enums. Scipp-free; runtime-side converter module.
  • Generic workflow models: monitor, detector, timeseries.
  • Per-instrument files relocated to schemas/instruments/<inst>/ by filesystem convention.
  • Identity simplification: WorkflowId → (instrument, name, version); WorkflowSpec.group; params → inputs.
  • Translation-layer convention: apply_params / dynamic_keys free functions.
  • Dependency-boundary CI: schemas/ imports pydantic + stdlib only.
  • Reduced-schema JSON emission on the status topic.

Out Deliberately not now

  • No ess.schemas extraction. Stays inside ess.livedata; migration to ess.reduce.schemas is a separable follow-up.
  • No composition-root refactor. Source-partitioning framing guides reduced-schema design, but the five-concern factory decomposition is separate Tier A work.
  • No cross-repo type sharing with technique packages. Per-technique submodules are opt-in, when maintainers engage.
  • No StreamProcessorWorkflow move to ess.reduce.
  • No cross-cutting transforms (detector views, histogramming) moving to ess.reduce.
  • No sciline-annotation shortcut on param models.
  • No wire-export of pydantic validators. Validators are Python-side; JSON Schema carries structure + bounds; backend validates authoritatively.
  • No publication as a PyPI package.

5 — Stage 0: in-place surgery

Non-1:1 semantic changes to WorkflowId and WorkflowSpec performed before any files move, so Stage 1's relocation is mechanical. Four independent commits on this branch; packaging (single PR vs. split) decided when the branch is ready.

0a — Route services without reading WorkflowId.namespace

JobManager.create currently rejects jobs where workflow_id.namespace != instrument.active_namespace (set by ServiceFactory at startup). Replace with an explicit service tag owned by the registering factory.

  • WorkflowFactory.register(...) gains a service: str argument.
  • Factory keeps a service mapping WorkflowId → str.
  • JobManager consults factory.get_service(workflow_id) and compares against the running service name.
  • active_namespace stays in place but becomes unused by routing; removed in 0d once nothing else reads it.
0b — BackendStatus.service_name replaces .namespace

BackendStatus publishes namespace; the dashboard uses it for the worker-identity key and status-widget label. Rename the wire field to service_name; update service_registry.py key composition and backend_status_widget.py display.

Wire break: producers and consumers land in the same commit — no compatibility shim.
0c — Add WorkflowSpec.group, rename params → inputs

Introduce WorkflowGroup as a frozen pydantic model providing display metadata:

class WorkflowGroup(BaseModel, frozen=True):
    name: Literal["data_reduction", "monitor_data", "detector_data", "timeseries"]
    title: str
    description: str = ""

Canonical instances as module-level constants (REDUCTION, MONITORS, DETECTORS, TIMESERIES); specs reference them. WorkflowSpec gains group: WorkflowGroup (required, no default) and renames params → inputs (and WorkflowConfig.params → WorkflowConfig.inputs for consistency).

6 — Stage 1: carve out schemas/

Everything generic; no instrument touched yet. Ends with every load-bearing type in place and a green repo, behind re-export shims so existing instrument code keeps importing from old paths.

Pre-migration audit

For every type that is a candidate to move into schemas/ — WorkflowSpec, param models (including DetectorViewParams and per-instrument subclasses), output models, InstrumentSpec / ComponentSpec — check its field types for scipp or callable leakage. Anything that fails decomposes before the file moves: move the offending field onto a runtime-side helper, keep only declarative data on the schema type.

Note: runtime-side config types (LogicalViewConfig, SpectrumViewSpec) hold callables and scipp types by design — they are inputs to factory/registration logic, not part of the wire surface. They stay where they are.

ArraySpec replaces sc.DataArray output templates

Current *Outputs models declare sc.DataArray with default_factory lambdas producing zero-sized arrays. The schema information consumers need (dims, unit, coord units) is only reachable by constructing a DataArray and reading it back.

class PowderReductionOutputs(WorkflowOutputsBase):
    focussed_data_dspacing: sc.DataArray = Field(
        default_factory=lambda: sc.DataArray(
            sc.zeros(sizes={'dspacing': 0}, unit='dimensionless'),
            coords={'dspacing': sc.zeros(sizes={'dspacing': 0}, unit='angstrom')},
        ),
        title='I(d)',
    )
@dataclass(frozen=True, slots=True)
class CoordSpec:
    unit: str | None = None

@dataclass(frozen=True, slots=True)
class ArraySpec:
    dims: tuple[str, ...]
    unit: str | None = None
    coords: Mapping[str, CoordSpec] = field(default_factory=dict)

class PowderReductionOutputs(WorkflowOutputsBase):
    focussed_data_dspacing: ArraySpec = Field(
        default=ArraySpec(
            dims=('dspacing',),
            unit='dimensionless',
            coords={'dspacing': CoordSpec(unit='angstrom')},
        ),
        title='I(d)',
    )

Consumer impact: WorkflowSpec.get_output_template() returns ArraySpec; find_timeseries_outputs becomes len(spec.dims) == 0 and 'time' in spec.coords; plotter auto-selection reads dims/coord units directly; sites that need an empty sc.DataArray at runtime use array_spec_to_dataarray(spec).

Parameter models go scipp-free

Today's parameter_models.py carries get_start(), get_stop(), get_edges(), range() methods returning sc.Variable / sc.scalar. Field declarations themselves are pure (float, int, str, enum). Strip the scipp-returning methods and relocate them as free functions.

# Methods pull scipp into the schema layer
class WavelengthRange(BaseModel):
    start: float
    stop: float
    unit: WavelengthUnit

    def range(self) -> tuple[sc.Variable, sc.Variable]:
        return sc.scalar(self.start, unit=self.unit.value), ...

# Usage
w1, w2 = params.wavelength.range()
# Pure pydantic, no scipp
class WavelengthRange(BaseModel):
    start: float
    stop: float
    unit: WavelengthUnit

# Runtime-side converter (parameter_conversions.py)
def range_to_scalars(m: WavelengthRange) -> tuple[sc.Variable, sc.Variable]:
    return sc.scalar(m.start, unit=m.unit.value), sc.scalar(m.stop, unit=m.unit.value)

# Usage
w1, w2 = range_to_scalars(params.wavelength)

Mechanical; ~20 call sites.

InstrumentSpec / ComponentSpec and the runtime split

Today's Instrument bundles schema data (name, title, description, detector/monitor names, component metadata, workflow specs) with runtime machinery (transform bindings, WorkflowFactory, stream/Kafka wiring, load_factories()). Split:

class InstrumentSpec(BaseModel):
    name: str
    title: str
    description: str
    detectors: dict[str, ComponentSpec]
    monitors: dict[str, ComponentSpec]
    workflows: dict[WorkflowId, WorkflowSpec]

Runtime Instrument holds an InstrumentSpec reference plus the runtime pieces. No exported discovery API; the runtime Instrument is built by setup_factories() as today.

Translation-layer convention

Per workflow impl, co-located with factory registration:

def apply_params(pipeline: sciline.Pipeline, params: DreamPowderWorkflowParams) -> None:
    pipeline[dream.powder.types.DspacingBins] = edges_to_variable(
        params.dspacing_edges, dim='dspacing', unit='angstrom'
    )
    # ...

def dynamic_keys(params: DreamPowderWorkflowParams) -> dict[str, sciline.typing.Key]:
    return {params.detector_source: NeXusData[NXdetector, SampleRun], ...}

Registration attaches apply_params and dynamic_keys alongside the WorkflowSpec and the sciline pipeline.

Scope note: a formal WorkflowImpl bundle (pipeline + accumulators + target_keys + apply_params + dynamic_keys as one dataclass) is Tier A follow-up. Near-term: documented convention.

Reduced-schema JSON emission

Each service emits, on its existing status topic:

  • Its service name (the one that replaced namespace in Stage 0b).
  • Per registered WorkflowId, the reduced JSON Schema: the param model minus fields filled by deployment pins or job-start routing.

Field partitioning

Pinned

Fields set by ops config at service startup. Example: calibration file path, NeXus geometry filename. Backend authority; UI never sees.

Job-start routed

Fields resolved from a job-launch selection — "which detector source is the mantle for this run". Filled by UI orchestration, not by the parameter form.

User-visible

Everything else. This is what appears in the JSON Schema on the wire.

No composition-root refactor is needed to land emission. The pinned/routed subset can be enumerated per workflow-impl registration in the near term (a list of field names excluded from emission). Full composition-root generalisation is Tier A follow-up.

8 — Later: migration to ess.reduce

Scope: explicitly out of this branch. Noted here so the future migration doesn't require a redesign.

Framework-only types (WorkflowSpec, WorkflowId, ArraySpec, InstrumentSpec, ComponentSpec, common parameter building blocks) eventually migrate into ess.reduce.schemas with the same pydantic+stdlib submodule-level boundary. Cross-repo, but within the same maintainer group — organisationally cheap.

Concrete trigger for the migration

Before a second instrument joins Stage 2/3 with stable schemas. Don't let "for now" become forever — otherwise the migration never happens and the types calcify in ess.livedata.

What stays where

Moves to ess.reduce.schemas

  • Framework types: WorkflowSpec, WorkflowId, ArraySpec, InstrumentSpec, ComponentSpec.
  • Common parameter building blocks (bin edges, ranges, unit enums).
  • Submodule-level pydantic+stdlib-only boundary.

Stays in ess.livedata

  • Per-instrument specialisations (DreamPowderWorkflowParams, …).
  • Stream mapping, Kafka plumbing, accumulators.
  • Workflow registrations, factories.
  • Composition root (when it exists).

9 — What could force reconsideration

Falsification conditions. The direction is worth pursuing only if it remains reversible on evidence. Each is a concrete signal, not a hypothetical.

ReversalThe second consumer never materialises

If 18 months after consolidation no technique-package ipywidget UI and no NICOS integration is pulling from the schemas, migration to ess.reduce is speculative. Consolidate, stay in ess.livedata, move on.

ReversalGraph-reshape discipline fails

If bespoke factories keep growing wf.insert(...) because upstream maintainers won't absorb category-C PRs, the shadow-algorithm rot wins and cross-compute unification is organisationally dead. Call it by name and stop planning around it.

TightenVersion mismatches become routine post-migration

Tighten backward-compat discipline, or reverse the migration to ess.reduce.

Scope-backTechnique-package maintainers reject per-technique schemas submodules

Cross-compute unification stays at zero; vocabulary is livedata-local. Still a coherent outcome.
