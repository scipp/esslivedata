Unified workflow-spec surface across ESS data pipelines and UIs

One vocabulary — WorkflowSpec, parameter models, output schemas — declared once, consumed uniformly by every UI (livedata dashboard, ipywidget UIs, CLI, NICOS) and satisfied uniformly by every compute backend (streaming, batch). This document makes the tradeoffs of that direction legible enough to adopt it, modify it, or reject it on what it actually costs.

1The problem — what's broken today

Six contexts run or present reduction workflows: streaming, batch, ipywidget UIs, the livedata dashboard, CLI, and external frontends (NICOS). Each answers "what workflows exist, what parameters, what outputs?" by reaching into its own, independently-drifting source.

Parameter defaults drift
DREAM d-spacing default in the livedata dashboard is not guaranteed to match ess.powder's notebook UI. Same workflow, different defaults.
Output-shape disagreements
I(d) from livedata vs. ess.powder can carry different coord units, dim names. Silent reconciliation required downstream.
Validation duplicated & drifts
"Must be positive" declared in one place, copied with minor variation elsewhere.
Duplicated UI form logic
Every UI builds its own form layer from scratch — no shared spec to render against.
NICOS has no path in
No single source of truth for "workflows available on DREAM right now" that out-of-process frontends can query.
Triple-edit contribution overhead
Adding a parameter to a powder workflow: edit technique package, edit livedata, edit ipywidget UI. Three changes for one concept.

Not pathological today (teams are close, duplication small) — pathological eventually (surface grows super-linearly, reconciliation projects get deprioritised).

2What a workflow spec is (and is not)

A workflow spec is vocabulary — a declarative description of a pipeline's interface. It answers four questions: identity, inputs, outputs, display metadata. The separation below is load-bearing: anything that pulls scipp, sciline, or a UI toolkit into the spec forecloses a consumer.

A spec IS

  • Identity — instrument, technique, version; unambiguous reference across UIs & releases
  • Inputs — typed parameters (ranges, defaults, units) + data sources (streams or NeXus groups)
  • Outputs — structural shape per result: dim names, unit, coord names/units. Schema, not arrays.
  • Display metadata — titles, groupings, plot hints (UIs need them; nowhere sensible else to live)

A spec is NOT

  • The algorithm — sciline graph, scipp transforms stay with technique packages
  • Runtime machinery — Kafka topics, accumulators, job scheduling
  • UI rendering code — forms, plots, widgets are consumers of the spec
  • Fine-grained data layout — bin-edge/center, dtype, chunk shape

Key design move: replace sc.DataArray output factories (current livedata) with a shape-only schema (ArraySpec-like) that constructs no arrays. This is what lets the spec layer use pydantic + stdlib only — the dependency cap that keeps it cheap to share.

3The goal — two unification axes

Cross-UI: same spec drives livedata dashboard, ipywidget UI, CLI, NICOS. Cross-compute: same spec describes streaming (esslivedata) and batch (technique package on NeXus file). Each cell consumes or implements the same spec. Today-empty cells become cheap because integration cost collapses.

4The ecosystem today

Stated as facts, not assuming any direction.

Players

  • ess.livedata — streaming + dashboard. Defines its own spec types (WorkflowSpec, parameter models). Carries sc.DataArray factories, pydantic validators, instrument subclasses. Specs, factories, Kafka adapters intermixed.
  • Technique packages (ess.powder, ess.sans, ess.reflectometry, ess.diffraction, ess.spectroscopy) — reduction algorithms as sciline pipelines. Each has own parameter-declaration convention. UIs ad-hoc.
  • ess.reduce — cross-cutting reduction primitives. Owns compute; does not today own vocabulary.
  • scippneutron — low-level neutron scipp layer.
  • NICOS — instrument control. Not today a Python consumer of any of the above.

Seams where drift lives

  • DREAM powder parameters exist both in the technique package and as livedata's DreamPowderWorkflowParams. Kept in sync manually.
  • Output shapes disagree: livedata's *Outputs with sc.DataArray factories vs. technique-package sciline returns.
  • Validators live in livedata only — technique package finds out about ranges when the notebook crashes.
  • Display metadata (titles, groups) livedata-only; ipywidget UI reinvents.

5Architectural direction under discussion

A three-way split with a deliberate choice about where vocabulary lives. Three structural claims sit inside this picture — each individually evaluable.

UI surfaces
livedata dashboard · ipywidget UIs · CLI · NICOS (future)
▼ consumes specs ▼
ess.schemas — the shared vocabulary
pydantic + stdlib only. No scipp, no sciline, no UI toolkit.
  • WorkflowSpec, WorkflowId, InstrumentSpec, ComponentSpec
  • Parameter models, output shape schemas (ArraySpec), display metadata
  • Possibly: per-instrument / per-technique subpackages (see Decision 1)
▼ implemented by ▼
Streaming backend — ess.livedata
  • Kafka source/sink, accumulators, orchestration
  • Live-only parameter overlays
  • Dashboard
  • ↓ uses ess.reduce (transforms)
Batch backend — ess.<technique>
  • sciline over NeXus files
  • Technique-specific algorithm
  • ↓ uses ess.reduce (transforms)

Three load-bearing claims inside this picture

CLAIM 1
Vocabulary is separable from compute
A spec can answer the 4 questions with pydantic + stdlib, no scipp. Empirically testable. The output-shape redesign is the hardest part.
CLAIM 2
Streaming is an adapter over shared compute
Cross-cutting data-shaping (detector views, histogramming, focus, ROI) belongs in ess.reduce. What stays in livedata is the reactive half: accumulators, Kafka, orchestration.
CLAIM 3
Technique packages become spec-conformant
Technique packages declare workflows against the shared vocabulary (or it mirrors them). Workflows become first-class, catalog-wide entries.

6Why this direction — and why it's hard

Symmetric honesty. Every upside has a cost that accrues to somebody. Toggle to compare side-by-side or focus on one.

Upsides

One vocabulary, many consumers
n×m problem collapses to n+m. One changelog, one compatibility story.
Technique packages gain a UI path
Any UI built against the vocabulary works for any technique package conforming to it.
Livedata gains reduction correctness for free
Livedata calls ess.powder (or shared ess.reduce transforms) directly. "Live differs from batch" becomes structurally hard.
Parallel interface declaration in livedata disappears
Ongoing manual sync burden ends. The drift-class bug ceases to exist, not because it's patched, but because the surface permitting it is gone.
Replay / stream-from-file become natural
Both backends speak the same spec. No new glue.
NICOS integration becomes possible
Single source of truth → out-of-process consumers have something to query. Today they do not.
Contribution overhead drops
Edit once; every UI picks it up; both backends see the new field.
Testability improves at every layer
Vocabulary testable in isolation (pydantic + stdlib). ess.reduce testable as batch transforms. Streaming testable as plumbing.
Design discipline is forced
Versioning, validator semantics, streaming-vs-batch boundaries stop being perpetually deferred.

Downsides

Beam-time fix latency
Most concrete cost. 2am validator fix: today ~15min in livedata. Post-unification: cross-repo PR, release cut, bump, deploy. Hours to days.
Version-mismatch failures
Backend v1.2 + dashboard v1.3 → ghost inputs; reverse → missing required fields. Didn't exist before.
Rollback asymmetry
Rolling livedata back while vocabulary stays may be unsafe; the reverse usually is. On-call needs a playbook that doesn't exist.
Cascading upstream evolution
Every non-trivial technique-package change is now a coordinated release train across 2–3 repos. Multiplied by 5–6 technique packages.
Breaking-change blast radius
DREAM-only concept change in shared vocabulary implicitly obligates every instrument/technique to re-verify.
Ownership dilution
Three-team bug = who owns, reviews, releases, deploys? Fails exactly under pressure.
Release-cadence coupling
Livedata ships when the slowest of {vocab, reduce, technique pkgs, livedata} is ready.
Dev-loop friction
Editable installs across multiple checkouts. Devcontainer that worked in one repo no longer end-to-end.
Cognitive overhead for contributors
Which of {vocab, reduce, livedata, technique pkg} owns this bug? Onboarding gets harder before it gets easier.
Each technique team becomes a stakeholder whose priorities matter
Today livedata decides in isolation. Post-unification, "let's change this" is a cross-team conversation.

7Fourteen decisions

Each must be made or consciously deferred. Options compound but are separable. Click any card to expand. The recommendation tilt per card is non-binding — it's the coherent stance reachable today given the rest.

1
Vocabulary scope — what lives in ess.schemas?
Framework only? + generic params? + instrument specializations? + technique specializations?
 Tilt: (c) → aim (d)
a
Framework only
Just cross-cutting types. No home for MonitorDataParams shared by livedata + ess.powder + NICOS → defeats unification.
b
Framework + generic params
Minimum that delivers cross-UI claim. Technique-specific specializations stay with technique packages.
c
Framework + generic + instrument subpackages tilt
Matches current code shape. Instrument specializations (e.g. DreamMonitorDataParams) first-class in vocabulary. Full cross-UI benefit; instrument-local changes become vocabulary changes.
d
+ technique subpackages (endpoint)
Max unification. Architecturally clean; organizationally aggressive — requires technique-package maintainers to accept shared vocabulary as source of truth. Not a starting position.
Recommendation Start at (c). Aim at (d). (d) requires organizational alignment not yet in place.
2
Physical home & ownership — where does the vocabulary live?
scipp/ess monorepo? separate repo? in-repo for now?
 Tilt: (d) now → (a/c) later
a
In scipp/ess monorepo as new subpackage
Proximity to technique-package maintainers. Monorepo release cadence may be incompatible with beam-time ops.
b
In monorepo with separately-scoped ownership
Same physical location, different reviewer pool. Only helps fix latency if vocab-owning team has cadence autonomy.
c
Separate repository
Cleanest on paper, operationally most complex. Version matrix, dependency bump PRs, rollback asymmetry all show up.
d
Inside ess.livedata for now; extract once 2nd consumer exists tilt
Enforce the dependency boundary (pydantic + stdlib) in-repo. Prove the shape. Extract when payoff is concrete. Fix latency stays at deploy speed.

Worst option: "extract early because it will be harder later" — extraction gets no easier if the boundary is unclear; extraction gets much easier if the boundary has already held under real use.

3
Authority for interface evolution — who decides what changes?
The single question most likely to determine whether unification is sustainable. Also the most easily deferred.
 Responsiveness > nominal authority
a
Technique packages lead
Vocab mirrors what technique pkgs publish. Every non-trivial change is a two-release dance. Vocab becomes lagging.
b
Vocabulary leads unrealistic
Vocab declares contract; technique pkgs conform. Inverts authority arrow — domain expertise is in technique pkgs. No leverage over non-livedata users.
c
Interfaces only in vocabulary (algorithms with technique pkgs)
Architecturally correct. Requires organizational alignment. Fatal risk: if vocab can't keep up, shadow interfaces form and unification dies quietly.
d
Instrument teams own their slice
Matches how ESS is actually organized. Crosscuts technique ownership (most instruments use multiple techniques). Works if coordinated; a mess otherwise.
Generative observation The authority question isn't just "who" but "how fast". Same process with hour-scale reviews is survivable; with week-scale reviews it's fatal. The real decision is responsiveness commitments.

Factory? adapters? callbacks? no contract?
 Tilt: (a)
a
Spec + factory tilt
Generalization of livedata's current factory. WorkflowFactory lookup by WorkflowId. Each backend owns its factory; specs key the lookup. Minimal vocabulary surface.

6
Presentation variants across UIs
Beam-time operator sees few knobs; batch analyst sees all. Three sub-problems easily conflated.
 Tilt: three mechanisms

Three sub-problems — each wants a different answer:

Visibility filtering
Which params a UI surfaces. Pure rendering concern. "Basic / advanced / expert."
Deployment-pinned values
Params fixed by deployment (norm file, calib). UI cannot change regardless of visibility. Ops has authority.
Scientifically-different variants
"Live powder" vs. "batch powder" are genuinely different computations — different identity, not different presentation.

Open questions
  • Who owns visibility hints — technique authors (domain judgment) or vocab maintainers (cross-cutting schema)? Likely split: vocab defines tag values, authors assign.
  • Do deployment overlays participate in spec discovery? Does get_workflows() return overlay-applied, raw, or both?
  • Can visibility hints be overridden per UI? Override invites drift; no override is rigid.
  • UI behaviour for pinned values: "fixed by deployment" indicator vs. absence? Affects user trust.

Runtime-catalog discovery — how UIs learn effective specs
Two catalogs: schema (what ess.schemas declares) vs runtime (what a deployment actually serves, overlay applied).
 Distinction: commit now. Mechanism: deferrable.

Two catalogs by design: batch notebook cares about schema catalog (no deployment); in-process dashboard reads runtime catalog directly (imports livedata's Python state); NICOS / remote / web-app dashboard cares about runtime catalog but cannot import Python.

▸ Publication channel (out-of-process UIs only)

Status-topic announcement tilt

b
Dedicated catalog endpoint
More explicit; more infrastructure.
c
Heartbeats with embedded catalog
Simple; bandwidth-redundant.
▸ What is published — effective specs, or base + overlay?
e1
publish-effective (pinned fields removed)
Information-lossy: UI can't render "fixed by ops" indicator — field is simply absent.
e2
publish-reference + overlay richer

Open questions

  • If the dashboard goes from co-resident to decoupled (web-app), reuse NICOS-targeted mechanism or build its own?
8
Validator policy & wire format
Pydantic validators are Python callables. They don't round-trip through JSON.
 Tilt: (a)
a
Validators = Python-side enrichment only tilt
Out-of-process consumers reimplement validation or accept its absence. Validators evolve at deploy speed — no cross-repo release needed for a validator fix.
b
All consumers are Python
Preserves validator authority. Forecloses NICOS-over-wire or any non-Python consumer.
c
Dual-mode per validator (tagged)
Correct in principle; painful in practice. Every validator needs a tag that survives years of drift.
Why (a) Beam-time fix latency is concrete; NICOS-over-wire is hypothetical. Don't trade a concrete cost for a hypothetical benefit. Revisit only if the hypothetical becomes concrete.

8Dev & prod impact — today vs post-unification

Six scenarios on the dev side, six on the ops side. Use the filter to focus. Production scenarios assume extraction has happened — in-repo consolidation phase does not incur most of these.

Adding a parameter to a workflow DEV

TodayEdit DreamPowderWorkflowParams, factory, tests — one PR in livedata.
Post-unificationEdit shared vocab (possibly different repo), wait for vocab release, bump livedata, update factory. In parallel update ess.powder. Days to a week.

Fixing a validator during beam time DEV

Today2am one-line PR, on-call review, redeploy. ~15 min.
Post-unificationDepends entirely on validator policy (Decision 8). With (a): deploys at livedata speed. With (b): cross-repo PR + release + bump.

Adding a new instrument DEV

TodayNew subpackage under livedata's config/instruments/.
Post-unificationNew subpackage in vocab, possibly new transforms in ess.reduce, possibly new technique-pkg entries. 3-repo coordination exactly at the moment contribution matters most.

Technique package bumps reduction interface DEV

TodayLivedata may not even notice — has its own param model and entry point.
Post-unificationVocab tracks the signature directly. Routine cost doubles or triples per release. This is where the budget actually goes.

New contributor fixes a monitor-histogramming bug DEV

Todaygrep in livedata, find, fix, test, PR.
Post-unificationFirst figure out which layer — vocab? reduce? livedata? technique pkg? — set up editable installs, cross-layer tests, per-repo release flow. Hours of first-time overhead.

Cross-layer debugging DEV

TodayDashboard shows wrong values → trace through one codebase.
Post-unificationVocab (spec wrong?) → livedata (adapter wrong?) → ess.reduce (transform wrong?) → technique pkg (algorithm wrong?). Each boundary is where the bug hides. Needs layer-aware observability that doesn't exist today.

Rolling deploy during an active run PROD

TodayBackend + dashboard from same codebase → spec shapes agree by construction.
Post-unificationBackend v1.3 + dashboard v1.2 → ghost inputs. Reverse → validation failures on arrival. Mitigation: backward-compat discipline + tight windows.

Rollback during an incident PROD

TodayRollback is a livedata-only action. Usually safe.
Post-unificationLivedata rollback while vocab stays may be unsafe; vocab rollback almost never right. On-call needs a playbook. Rollback artifact must include vocab pin.

Validator change during production PROD

TodayShips with livedata push.
Post-unificationWith validator-policy (a): livedata-speed. With (b): vocab release — which during beam time may be operationally blocked. This is where operational flexibility lives.

Scheduled vocab release during beam time PROD

TodayN/A.
Post-unificationLivedata decides: pin now, wait, or skip. Requires vocab-owning team to know + respect ESS run schedules.

Cross-team on-call PROD

TodayOne person, merge rights on one repo.
Post-unificationUrgent fix may require merge rights on vocab and/or technique pkg. Either cross-team on-call (org commitment), bypass paths (op risk), or longer incident windows.

Observability across layers PROD

TodayLogs livedata-internal state only.
Post-unificationNeed to log: which vocab version produced this spec? which ess.reduce transform? which adapter wired them? If not built pre-extraction, the first post-extraction incident is also the first tracing attempt. Bad combination.

9What could force a reversal

Falsification conditions. The direction is worth pursuing only if it remains reversible on evidence. These are conditions, not predictions.

Second consumer never materializes
18 months after consolidation, no technique-pkg ipywidget UI and no NICOS integration is pulling from the vocab → extraction is speculation. Stay in-repo; move on.
Fix-latency incidents cluster around vocabulary changes
If on-call log shows vocab-side fixes dominate incidents post-extraction, the operational cost is real. Either validator-policy (a) is non-negotiable, or the direction is wrong for this reality.
Technique-package maintainers reject spec conformance
If, when asked to declare workflows against shared vocab, they push back on cadence, authority, or design — unification is organizationally infeasible. Fall back to vocab-used-only-by-livedata; accept no cross-compute unification.
ess.reduce cannot absorb the transforms
If maintainers see detector-view logic as livedata-specific clutter, cross-compute unification-in-compute is blocked. Livedata keeps transforms; unification stays at the vocab layer only.
Version mismatches become routine
More than one per quarter despite CI discipline → pinning strategy is wrong. Tighten or reverse.
Dev-loop friction is the top contributor complaint
If new contributors consistently fail to land changes because of cross-repo coordination, the cost is higher than anticipated.
Cross-team coordination does not form
Vocab-owning team unresponsive → validator fixes take days, interface changes take weeks, livedata maintainers work around vocab rather than through it. Unification failing quietly. Detect and reverse.

10Open questions for explicit resolution

Questions this document cannot answer alone. They need the relevant stakeholders in the room before extraction starts, ideally before consolidation ends.

Do technique-package maintainers (powder, SANS, reflectometry, diffraction, spectroscopy) accept that their workflows become first-class entries in a shared vocabulary? If not in full, what subset?
Does the ess.reduce maintainer group accept absorbing detector-view, monitor-histogramming, and related cross-cutting transforms currently in livedata? With what timeline?
Is there a second concrete consumer of the vocabulary today (a technique-pkg ipywidget UI in development, a NICOS integration with real requirements)? If not, plausible timeline to one? This is the primary gate on extraction.
scipp/ess monorepo release cadence — and process for triggering an out-of-cycle release for an urgent fix? Determines whether validator policy (a) or (b) is operationally viable.
Does NICOS need JSON-Schema exports or Python-class imports? Answer determines whether validator-policy is a real decision or a forced choice.
Do technique packages treat workflow interfaces as public APIs with deprecation discipline, or as internal-to-livedata structures free to change? Determines whether authority-option (a) is expensive or ruinous. Needs a changelog survey, not a guess.
Who is responsible, post-extraction, for coordination costs across repos? If "the livedata maintainer absorbs it" — that's an explicit organizational ask; it should not be implicit.
Is the goal extraction, or is the goal internal separation with a clear exit path if demand appears? Different commitments, different cost structures. Easy to conflate.
What observability exists today to diagnose layer-crossing bugs, and what would need to exist before extraction is operationally safe? If answer is "none" and "substantial" — that work is on the critical path.
Can parts of the vocabulary be auto-generated from technique-package type annotations, so vocab tracks interfaces without requiring a separate release per change? Speculative but potentially consequential.

Recommendation tilt (non-binding)

A coherent stance reachable today, with the decisions above made explicit:

  1. Consolidate in-repo. Carve out the vocabulary subpackage inside livedata; enforce dependency boundary (pydantic + stdlib only); replace sc.DataArray output templates with shape-only schema. Hold the boundary with CI. Captures most internal benefit — testability, separation, forced design decisions — at none of the cross-repo coordination cost. Makes extraction materially easier later by proving the boundary holds.
  2. Extraction gated on a second concrete consumer, not speculation. Technique-pkg ipywidget UI or NICOS integration with real code + real requirements → begin extraction. Eighteen months without one → consolidation is the endpoint.
  3. Commit to validator-policy (a) — validators are Python-side enrichment, not wire contract. Beam-time fix latency is concrete; NICOS-over-wire is hypothetical. Revisit only if hypothetical becomes concrete.
  4. Cross-cutting transforms → ess.reduce as a separable plan with its own audit, requiring ess.reduce-maintainer buy-in. Don't let it block vocabulary work; don't let it be silently deferred by vocabulary work either.
  5. Vocabulary scope at 1(c): framework types + generic parameter models + per-instrument specializations. Not per-technique specializations yet. Per-technique absorbs into vocabulary only once technique-pkg maintainers are concrete participants, not proxies.
  6. Before any extraction, make explicit the responsiveness commitment from the vocab-owning team. Hour-scale for urgent fixes. Day-scale for routine. Week-scale for design. Without this, authority-option (a) is ruinous regardless of what the org chart says.
The failure mode to avoid is not extracted too early or extracted too late. It is extracted without deciding the validator policy, the responsiveness commitment, and the second-consumer gate. Those three questions, resolved, make the direction survivable whether extraction happens in a year or never. Left unresolved, any version of the direction fails under pressure.
